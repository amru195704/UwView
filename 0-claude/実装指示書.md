# UwView 再作成 実装指示書（Avalonia版 / Claude Code 向け）

## 0. この文書について

- 目的: かつて Vector で公開していた大容量テキストビューア **UwView**（最大2億行を高速閲覧）を、現在の環境で再作成する。
- **UIフレームワークは Avalonia UI**（Win/Mac/**Linux** を正式サポート、独自Skia描画）。**ネイティブ優先＋Web(WASM)はおまけ**の方針。
- この文書は **企画＋実装指示書**。実装・ビルド・コミットは Mac 側の Claude Code（VSCode/Rider）で行う。Cowork 側では設計のみ。
- 元アプリの位置づけ: 大容量・高速テキスト **ビューア**（エディタではない）。通常のエディタは100万行程度で開けなくなるが、UwView は最大2億行を実用的に閲覧できることが唯一無二の価値。RDB/XML ダンプ等で発生する巨大行数ファイルを「とにかく速く見る」ことに特化する。
- **UwView は Geo 製品群ではない。** GeoCore 等 Geo 資産への依存は持たせない。純粋な単体ユーティリティとして作る。

---

## 1. 再現スコープ（今回）

優先機能は次の2点。まずこの2点を確実に動かすことを最優先とする。

1. **超高速な巨大ファイル表示**（最重要コア）— 最大2億行クラスを省メモリ・高速に閲覧。
2. **文字コード自動判定** — BOM＋UTF-8 / Shift-JIS / EUC-JP の自動判別と手動切替。

今回スコープに含む: **マルチファイル・タブ（複数ファイルを切替表示、§4.7）**。
将来拡張（今回は必須でないが設計で穴をふさぐ）: 文字列検索・正規表現、行ジャンプ強化、行番号/折り返し切替。

## 3. アーキテクチャ全体像

巨大ファイルの鉄則は「**ファイル全体をメモリに載せない／全行を UI に載せない**」。4層で実現する。

```
┌────────────────────────────────────────────┐
│ UI 層（Avalonia 独自描画の仮想テキスト面）    │
│  - 画面に見えている数十行だけを Render で描く │  ← 独自描画＋仮想化
│  - スクロールバー＋ホイール/キーで位置制御    │
└───────────────┬────────────────────────────┘
                │ GetPageAt(byteOffset) / GetLine(lineIndex)
┌───────────────▼────────────────────────────┐
│ ドキュメント層（LineDocument）               │
│  - バイト位置/行番号 → テキスト を返す        │  ← オンデマンド＋LRUキャッシュ
└───────────────┬────────────────────────────┘
                │ 行/ページの byte 範囲を解決
┌───────────────▼────────────────────────────┐
│ インデックス層（SparseLineIndex）＋文字コード │  ← スパース索引（省メモリ）
└───────────────┬────────────────────────────┘
                │ Read(offset, length)
┌───────────────▼────────────────────────────┐
│ I/O 抽象層（IByteSource）                     │
│  - Desktop: MemoryMappedFile（mmap）★優先    │
│  - Browser: Blob.slice（JS interop）おまけ    │
└─────────────────────────────────────────────┘
```

設計の肝は2つ。

- **I/O抽象 `IByteSource`**: `long Length` と `int Read(long offset, Span<byte> buffer)` だけの薄い抽象。ネイティブは mmap、WASM は `Blob.slice` で実装を差し替える。上位3層（索引・ドキュメント・UI）は I/O 実装を一切知らない。
- **スパース行インデックス**: 2億行分のオフセットを全部持つと `long[]`＝約1.6GB になり本末転倒。そこで **N行ごと（既定256行）に1つだけ**バイトオフセットを記録する。索引サイズは 2億 ÷ 256 × 8byte ≒ **約6MB**。任意行アクセスは直近チェックポイントから最大 N−1 行ぶんの改行を数え直すだけ（数KB〜数十KBの前方読み＝一瞬）。当時の低メモリ機で5000万行を実現し、現行機なら2億行まで同じ理屈でスケールする。

mmap（またはBlob.slice）と索引は競合せず組み合わせる。I/Oは実バイトの読み手段、索引は行→バイト位置の対応表。

### 3.1 起動時の3段階（プログレッシブ・オープン）

「待たされない」体験のため、**2つのナビゲーションモードを段階的に切り替える**。

1. **起動直後＝ページモード（索引なしで即閲覧）**
   - `IByteSource` でファイルを開き、先頭サンプルから文字コードだけ判定（一瞬）。
   - 索引を待たず、**バイト位置ベース**で「あるバイトオフセットから1画面ぶん」を描画。
   - ページ先頭が行の途中で切れないよう、**指定オフセットから次の `\n` までを飛ばして行境界に揃える**（前方スキャンのみ、索引不要）。
   - スクロールバーは**行数ではなくバイト位置（0〜ファイル長）**。ステータスは「M行目」ではなく「◯◯% / バイトオフセット」。「ファイルの50%へ」も即ジャンプ可（`less` が巨大ファイルを開いた瞬間と同じ挙動）。
   - **ページアップ**は後方スキャン（前の `\n` を遡る）で行頭を求める。mmap/Blob いずれもランダム読みで軽い。長大行（画面幅超）は横スクロール/クランプ。
2. **裏でスパース索引を構築開始**（§4.3、背景タスク、進捗＋キャンセル）。ページモードのまま閲覧継続。WASM では Web Worker 相当（別スレッド/非同期）へ逃がしてUIを固めない。
3. **索引完了＝行モードへ昇格**
   - スクロールバーが**行数基準**に。総行数表示・行番号ジャンプが有効化。
   - 切替時は**現在のバイトオフセットを最寄り行番号に変換**して表示位置を連続させ、継ぎ目を見せない。

> UI 実装の含意: 「ページモード」「行モード」の2状態を持つVMにし、スクロールバーの意味（バイト⇔行）とステータス表示を状態で切替。ドキュメント層は `GetPageAt(byteOffset)` と `GetLine(lineIndex)` の両APIを持つ。

## 4. コンポーネント設計

### 4.1 IByteSource（I/O抽象）

責務: 実バイトのランダム読みを、OS/環境非依存の最小APIで提供する。

```csharp
public interface IByteSource : IAsyncDisposable
{
    long Length { get; }
    // offset から buffer.Length バイト読み、実際に読めたバイト数を返す（末尾で不足可）
    int Read(long offset, Span<byte> buffer);
}
```

実装:
- **MmapByteSource（Desktop・★優先）**
  - `MemoryMappedFile.CreateFromFile(path, FileMode.Open, null, 0, MemoryMappedFileAccess.Read)` で開く（**mapName は必ず null**＝Unix非対応のため）。
  - 読む付近だけ「ウィンドウ・ビュー」（offset,size 指定の `MemoryMappedViewAccessor`）を張り、`SafeMemoryMappedViewHandle.AcquirePointer` の unsafe ポインタで高速コピー/スキャン。64bitなら全体ビューも可だがウィンドウ方式が素直。
  - 読み取り専用なのでマップ中はファイルロック（ビューアなので許容）。外部から途中で切り詰められるとアクセス時に落ちうる（UnixではSIGBUS）。既知リスクとして明記。
  - **代替**: 実装簡略化時は `System.IO.RandomAccess.Read(SafeFileHandle, ...)`（スレッドセーフ）でフォールバック可。
- **BlobByteSource（Browser・おまけ）**
  - Avalonia の `IStorageProvider`（`TopLevel.StorageProvider`）でファイル選択/ドロップ→ JS 側に `File`(Blob) を保持。
  - `Read` は **JS interop で `blob.slice(offset, offset+len).arrayBuffer()`** を呼び、結果を .NET 側 `Span` にコピー。**全体をメモリに載せない**。
  - ブラウザのファイルI/Oは本質的に非同期。同期 `Read` を satisfy するため、UIスレッドから直接叩かず「索引構築・先読みは非同期タスク側で行い、結果をキャッシュしてから描画する」設計にする（描画スレッドで await しない）。必要なら `IByteSource` に `ReadAsync` も足し、WASM は async 経路、Desktop は同期経路を使う二本立てにしてよい。
  - WASM32 は約4GBメモリ上限。索引(~6MB)と可視ウィンドウのみ WASM ヒープに置く。

> まず **MmapByteSource だけ**実装してデスクトップを完成させ、Browser head は後追い。

### 4.2 EncodingDetector（文字コード自動判定）

責務: ファイル先頭サンプル（既定 256KB）から使用エンコーディングを推定。

前準備:
- `System.Text.Encoding.CodePages` を参照し、起動時に `Encoding.RegisterProvider(CodePagesEncodingProvider.Instance);`（Shift-JIS=932 / EUC-JP=51932 用）。

判定手順（この順）:
1. **BOM 判定**: `EF BB BF`→UTF-8(BOM) / `FF FE`→UTF-16LE / `FE FF`→UTF-16BE /（余裕あれば `FF FE 00 00`→UTF-32LE）。
2. **BOMなし→UTF-8妥当性検査**: 厳格UTF-8（`new UTF8Encoding(false, throwOnInvalidBytes:true)`）でデコード試行。成功かつマルチバイト含む→UTF-8。全ASCII→UTF-8扱い。
3. **UTF-8失敗→Shift-JIS/EUC-JPをスコアリング**:
   - Shift-JIS: 先頭 `0x81–0x9F|0xE0–0xFC`＋次 `0x40–0x7E|0x80–0xFC`。
   - EUC-JP: 2バイト両方 `0xA1–0xFE`（半角カナは `0x8E`+`0xA1–0xDF`）。
   - 妥当2バイト列が多く不正バイトが少ない方を採用。同点は **Shift-JIS 優先**。
4. 決め手なし→既定 **Shift-JIS**。後から手動切替可（§4.6）。

出力:
```csharp
public sealed record DetectedEncoding(Encoding Encoding, string DisplayName, bool HasBom, int BomLength);
public static class EncodingDetector { public static DetectedEncoding Detect(IByteSource src, int sampleBytes = 256*1024); }
```
`BomLength` は本文開始オフセット（索引の起点）に使う。

### 4.3 SparseLineIndex（スパース行インデックス）

保持データ:
```csharp
long fileLength; int blockLines /*=256*/; List<long> checkpoints; long totalLines; int bomLength; NewlineStyle newline;
// checkpoints[k] = (k*blockLines) 行目の開始バイトオフセット
```

改行の扱い:
- `\n`(0x0A) を行区切りとして数える。`\r\n` は `\n` 側で1行、行取得時に末尾 `\r` を除去。
- `\r` 単独（旧Mac）モードも持つ。先頭サンプルで改行スタイルを推定して切替。既定は LF/CRLF。
- 末尾に改行が無い最終行も1行として数える。

構築（背景タスク、大バッファ順次読み）:
```
offset=bomLength; lineNo=0; checkpoints=[bomLength]
1MBずつ Read しながら '\n' を数える:
    if b=='\n': lineNo++; if lineNo % blockLines == 0: checkpoints.Add(pos+1)
    定期的に progress = bytesRead/fileLength を報告
totalLines = lineNo + (末尾に改行なしの残りがあれば 1)
```
- `IProgress<double>` と `CancellationToken` を受ける。数GBでも1回の順次読みで完了。
- 索引は約6MB / 2億行（チェックポイント約78万件）。`List<long>` で十分。
- （将来）プログレッシブ: 構築途中で `totalLines` を逐次公開し既索引範囲を先に行モード化。今回は「ページモードで即閲覧→索引完了で行モード昇格」で可。

### 4.4 LineDocument（オンデマンド取得）

`GetLine(long lineIndex)`:
```
k = lineIndex / blockLines; start = checkpoints[k]; target = lineIndex % blockLines
IByteSource.Read で start から前方走査し '\n' を target 個通過 → lineStart
さらに次の '\n' or EOF まで → lineEnd
bytes = [lineStart..lineEnd); CRLFなら末尾\r除去
text = encoding.GetString(bytes)（長大行は表示用に先頭N文字クランプ＋「…」）
return text
```

`GetPageAt(long byteOffset, int rowsPerScreen)`（ページモード用, §3.1）:
```
byteOffset から次の '\n' へ行境界を揃える（先頭以外）
そこから rowsPerScreen 行ぶんを順次読み・デコードして返す
上方向は前の '\n' を遡って行頭を求める
```

実装注意:
- **LRUキャッシュ**: 直近アクセス行テキストを（例4096行）LRUで保持。高速スクロールのちらつき防止。
- **長大行**: 表示用に先頭 8192 文字程度でクランプし末尾に `…（省略）`。全文はコピー機能で別途（将来）。
- **タブ**: まずは等幅フォント描画でそのまま。タブ幅設定は将来。

### 4.5 仮想テキスト面（Avalonia 独自描画コントロール）

CollectionView 相当は使わず、**自前描画コントロール**を作る（2億行を最も確実・高速に扱える）。

- `TextView : Control` を作り `public override void Render(DrawingContext ctx)` を override。
- 状態: 先頭表示位置（ページモード=バイトオフセット / 行モード=先頭行番号）、行高（等幅フォントの実測）、可視行数＝`Bounds.Height / lineHeight`。
- 描画: 可視行だけを `LineDocument` から取得し、各行を `ctx.DrawText(...)`（高速化するなら `GlyphRun`）で描く。行番号列（行モード時）は右寄せ・淡色。
- スクロール: 縦 `ScrollBar` を隣に置き、`Maximum` をページモード=fileLength / 行モード=totalLines に設定。`Scroll`・マウスホイール(`PointerWheelChanged`)・キー（PageUp/Down, Home/End, ↑↓）で先頭位置を更新し `InvalidateVisual()`。
- 横スクロール: 長大行用に横 `ScrollBar` を任意で（初期はクランプ表示で可）。

> AvaloniaEdit は参考にするが、既定 `TextDocument` は全文をメモリに載せるため 2億行には使わない。上記の自前仮想面で実装する。

### 4.6 手動エンコーディング切替

- ツールバーに「文字コード」ドロップダウン（自動 / UTF-8 / UTF-8(BOM) / Shift-JIS / EUC-JP / UTF-16LE / UTF-16BE）。
- 変更時は `LineDocument.Encoding` を差し替え、LRUをクリアして再描画するだけ。**索引再構築は不要**（オフセットはバイト位置＝エンコーディング非依存）。ここがスパース索引の利点。

### 4.7 マルチファイル・タブ（DocumentSession）

責務: **1つの開いたファイル = 1つの `DocumentSession`**。タブUIはこのセッションを切り替えて表示する。

`DocumentSession`（`UwView.Core`、UI非依存）が抱えるもの:
```csharp
public sealed class DocumentSession : IAsyncDisposable
{
    public string FilePath { get; }         // 表示名の元
    public IByteSource Source { get; }       // mmap or Blob
    public DetectedEncoding Encoding { get; set; }   // 自動判定＋手動上書き
    public SparseLineIndex? Index { get; }   // 構築中は null / 進捗を公開
    public LineDocument Document { get; }     // GetPageAt / GetLine（自前LRU）
    // 表示状態（タブ切替で保持）
    public ViewMode Mode { get; set; }        // Page / Line
    public long TopByteOffset { get; set; }   // ページモードの先頭位置
    public long TopLine { get; set; }          // 行モードの先頭行
    public double IndexProgress { get; }      // 0..1
    // 将来: 検索結果(バイトオフセット列) / ブックマーク もここに持たせる
}
```

VM/UI:
- `MainViewModel` が `ObservableCollection<DocumentTabViewModel> Tabs` と `DocumentTabViewModel? Active` を持つ。`DocumentTabViewModel` は `DocumentSession` をラップ（表示名・索引中インジケータ・閉じるコマンド）。
- `TextView`（§4.5）は **Active タブのセッションを描画**。タブ切替＝Active差し替え＋`InvalidateVisual()` のみ（**状態保持で即時**、索引再構築なし）。
- タブ操作: 追加（開く／ドラッグ&ドロップ／複数選択で一括）、閉じる（**セッションを Dispose → mmap 解放・ファイルロック解除**）、並べ替え。
- **各タブは独立に背景索引**。ページモードのタブと行モードのタブが混在してよい。非アクティブタブの索引も継続してよい（負荷を見て一時停止可）。

リソース注意:
- Nファイルで N×（≈6MB索引＋mmapビュー）。各索引は小さいが、**mmap は各ファイルをロック**する。閉じたら必ず `Dispose`。多数タブ時は LRU キャッシュ合計サイズに上限を設けてもよい。
- WASM 版は各タブが別 Blob（JS側）を保持し同様に独立索引。

UI: 上部にタブストリップ（Avalonia の `TabControl` か自前タブバー）。各タブに「表示名＋索引中インジケータ（スピナ/％）＋閉じる×」。

> これは §11 の⑨から昇格した**本体機能**（今回スコープ）。§11 に残すのはタイムスタンプ・マージ等の発展形のみ。

## 6. 実装手順（この順で進める）

### フェーズ0: 足場（Desktopのみ）
1. `avalonia.xplat` でソリューション生成。まず Desktop head だけ残し、`global.json`・`jp.y4u.uwview`・net10・Nullable/ImplicitUsings を設定。
2. **素の状態で Win/Mac/Linux の Desktop ビルド＆起動**を確認（Macで `dotnet run --project UwView.Desktop`）。

### フェーズ1: コアエンジン（UI非依存で単体検証）
3. `IByteSource` と `MmapByteSource` 実装。
4. `EncodingDetector` 実装 → UTF-8/BOM付/Shift-JIS/EUC-JP サンプルで判定一致をテスト。
5. `SparseLineIndex` + `LineDocument.BuildIndexAsync` 実装。
6. `LineDocument.GetLine` / `GetPageAt` 実装（走査＋LRU）。
7. **xUnit 検証**: 10万行の既知ファイル（各行に行番号）で `GetLine(0)/末尾/ランダム` が正しい・`TotalLines` 一致。巨大ファイルで索引構築時間とメモリを実測。
   - 検証ファイル生成（Mac）: `seq 1 200000000 | awk '{print "line "$1" テスト行"}' > /tmp/huge.txt`（2億行＝十数GB。空き容量に注意。まず1000万行程度で機能確認→本番2億行で実測）
   - Shift-JIS版: `iconv -f UTF-8 -t SHIFT_JIS /tmp/huge.txt > /tmp/huge_sjis.txt`

### フェーズ2: UI（ページモード先行 §3.1）
8. `TextView`（§4.5）を実装し、**まずページモード**で即表示（mmap＋文字コード判定のみ）。スクロールバー=バイト位置、ステータス=％/オフセット。ホイール/キー/％ジャンプを動かす。
9. **裏で索引構築**（背景・進捗・キャンセル）。構築中もページモードで閲覧継続。
10. **索引完了で行モード昇格**: 現在バイトオフセット→最寄り行番号に変換して位置継続。スクロールバー/ステータスを行基準へ切替。行番号列表示・行番号ジャンプ有効化。
11. ツールバー（開く=`IStorageProvider`、文字コード切替、ジャンプ）＋ステータスバー（パス/総行数/現在行/エンコーディング/索引進捗）。

### フェーズ2.5: マルチファイル・タブ（§4.7）
- `DocumentSession` をコアに追加し、単一ファイル前提だった UI を **タブ集合＋Active** 構成へ移行。`TextView` は Active セッションを描画。
- 開く／閉じる（Dispose で mmap 解放）／切替（状態保持で即時）／ドラッグ&ドロップ追加。各タブ独立に背景索引。

### フェーズ3: 仕上げ（Desktop）
12. 2億行ファイルでスクロール追従・末尾ジャンプ・体感速度を確認。LRUサイズ調整。
13. 文字コード手動切替が**索引再構築なし**で即反映されることを確認。
14. README 整備、既知の制限を明記。

### フェーズ4: WASMおまけ
15. Browser head を追加。`BlobByteSource`（JS interop の `blob.slice`）を実装し `IByteSource` に適合。非同期I/Oは先読み/索引側に隔離しUIを固めない。
16. ブラウザでファイル選択→ページモード→索引→行モードが動くことを確認（Chromium系優先）。既知制限（毎回ファイル選択、パス直開き不可、索引はネイティブより遅い）を明記。

## 8. パフォーマンス目標・受入基準

| 項目 | 目標 |
|------|------|
| 起動〜表示 | ファイル選択から**即ページモード表示**（索引を待たない）。 |
| 索引構築 | 2億行（数GB）を1回の順次読みで完了。進捗表示・キャンセル可。 |
| メモリ | 索引は数MB（`blockLines=256` で ~6MB / 2億行）。ファイル本体は非常駐。 |
| 行アクセス | 任意行の `GetLine` が体感即時（数ms）。高速スクロールでもカクつかない。 |
| エンコーディング | UTF-8/UTF-8(BOM)/Shift-JIS/EUC-JP/UTF-16 を自動判定。手動切替で即修正。 |

受入テスト:
1. 10万行: 先頭/末尾/ランダム行が正しい・総行数一致。
2. Shift-JIS/EUC-JP/UTF-8(BOM) が自動で正しく表示。
3. 2億行: 起動直後にページモードで即閲覧でき、索引完了後に末尾(2億行目)へ行ジャンプして正しい行が見える。メモリ増加が数MB規模。
4. 文字コード手動切替が再索引なしで反映。
5. （おまけ）ブラウザ版で同ファイルがページモード→行モードで閲覧できる。

## 9. 既知のリスク・判断ポイント

- **WASMの制約**: mmap不可→`Blob.slice`。全I/O非同期→索引/先読み側に隔離。4GBメモリ上限→本体非常駐を厳守。毎回ファイル選択・パス直開き不可・索引はネイティブより遅い。→ **あくまでおまけ**。
- **`int` 上限**: 2億行は int 上限（約21.4億）内なので UI境界の int 丸めも問題なし。かつ本設計は自前描画方式で `IReadOnlyList.Count` に依存しないため、CollectionView 由来の件数上限も無い。APIは `long` 行番号で通し、約21億行超を将来狙う場合のみ描画/スクロール境界を全面 long 化。
- **改行スタイル混在**: サンプルで推定＋`\n`基準でおおむね吸収。CR単独主体は別モード。
- **文字コード判定の限界**: 短小/記号のみは推定困難。既定Shift-JIS＋手動切替で担保。
- **mmapのSIGBUS**: 外部からのファイル切り詰めでアクセス時に落ちうる。読み取り専用ビューアとして許容・明記。

## 11. 機能追加案（Phase3以降・競合調査反映）

出典: `0-claude/UwView競合調査・機能追加案.md`（2026-07-02）。競合調査の結論として、**「Win/Mac/Linux/Web クロス」×「日本語エンコード自動判定」×「2億行を索引なしで即時表示（ページモード）→行モード昇格」の3点を同時に満たすツールは現存しない**。この差別化を強めるための Phase3 候補を、本設計の各層への実装フックとともに以下に定義する。

> 位置づけ: §6 のフェーズ0〜4（Desktop コア完成→WASM）が済んだ後の拡張。既存4層（`IByteSource` / `SparseLineIndex` / `LineDocument` / `TextView`）を壊さず“乗せる”ことを原則とする。

### 🔴 優先度高（コア強化・最大の差別化）

**① 文字列検索（背景スレッド・バイト列直接スキャン）**
- mmap のバイト列をマルチスレッドでスキャンし、エンコーディング変換後に正規表現マッチ。ヒットは**バイトオフセットのリスト**で保持し、ヒット行へ即ジャンプ。
- 実装フック: `SparseLineIndex`/`LineDocument` とは独立した `SearchService`（`IByteSource` を共有）。**索引構築と別スレッド**で走らせ、ページモード中でも検索可能に。ヒットのバイトオフセット→（索引完了後は）行番号へ変換。
- 発展: Boolean 式（`error AND timeout`）対応で本格ログ解析へ。

**② キーワードハイライト（複数色）**
- 複数キーワードを色分けして同時ハイライト。
- 実装フック: `TextView.Render` の行描画で、`DrawText` を部分文字列単位に分割して色替えするだけ（新レイヤ不要）。プリセット色（エラー=赤/警告=黄/情報=青）を用意しログ用途に直結。

**③ リアルタイム Tail（ファイル末尾追従）**
- `FileSystemWatcher` で更新検知、末尾表示中は自動スクロール。競合（LogExpert/LogViewPlus）の主力機能。
- 実装フック: 追記を前提に `SparseLineIndex` の**末尾チェックポイントを動的延長**（増分索引）。`IByteSource.Length` の変化を検出して差分だけ走査。mmap は再マップが要る点に注意（ビュー再取得）。

### 🟡 優先度中（使い勝手向上）

**④ ブックマーク**
- 任意行にブックマーク、前後ジャンプ＋一覧パネル。**バイトオフセットで保持**するのでエンコード切替後も有効。
- 実装フック: セッション保存（ファイルパス＋オフセットのペアをJSON）で再開を容易に。

**⑤ 行フィルタ表示（仮想フィルタビュー）**
- マッチ行のバイトオフセットリストを「仮想ドキュメント」として、マッチ行だけを別ビュー表示。元ファイルは不変。AND/OR 条件対応で差別化。
- 実装フック: `LineDocument.GetLine` をラップする `FilteredLineDocument`（マッチ行オフセット配列を保持）を作るだけで既存 `TextView` に乗る。①の検索結果を入力にできる。

**⑥ ミニマップ（スクロールバー横ヒートマップ）**
- ファイル全体を縦圧縮し、検索ヒット・ブックマーク位置を色点表示。2億行のどこに何があるかを一目で把握。
- 実装フック: 自前描画コントロールを活かした「UwViewらしい」見せ場。①②実装後に追加が自然。

### 🟢 優先度低（将来拡張）

**⑦ 構造化データプレビュー（JSON/CSV/XML）**: 「テキスト ↔ 整形済み」切替。巨大JSON/CSV閲覧向け。
**⑧ 行範囲エクスポート**: 指定行範囲をクリップボード/新規ファイルへ。`IByteSource.Read` のバイト範囲でそのまま実装可。
**⑨ タイムスタンプ・マージビュー**: （基本のマルチファイル・タブは **§4.7 で本体機能に昇格済み**。）複数ログをタイムスタンプでソートしてマージ表示する発展形。本格ログ解析への発展で実装コスト高。

### 推奨実装順（Phase3）

```
① 文字列検索（背景スレッド）
    ↓
② キーワードハイライト（複数色）＋ ⑥ ミニマップ（同時）
    ↓
③ リアルタイム Tail
    ↓
④ ブックマーク  →  ⑤ 行フィルタビュー
```

①②⑥が揃った時点で「クロスプラットフォーム・日本語エンコード自動判定・2億行対応の**ログ解析ビューア**」として明確なポジションが確立される。⑦⑧⑨は需要に応じて追加。

> 設計原則の再確認: 検索・フィルタ・ブックマークは**すべてバイトオフセット基準**で持つ（行番号ではなく）。これによりエンコード切替・索引未完了（ページモード）でも一貫して機能し、索引完了後に行番号へ写像できる。§4 の「バイト位置が一次、行番号は索引による写像」という土台をそのまま活かすこと。
